Motivation

[!img] ein bild von einem chatlog wo die dateiübertragung nicht funktioniert
- xmpp ist im großen und ganzen self-hosted von privatleuten -> keine Bandbreite Dateiübertragungen über Server abzuwickeln -> P2P klappt superduperselten # todo: was macht xmpp eigentlich standardmäßig für dateiübertragungen?

die Idee

[!img]...
  1. Dateien/Verzeichnisse für Kontakte freigeben
  2. Freigabelisten an alle Kontakte übermitteln
  3. als Empfänger: bekommene Dateilisten aggregieren, durchsuchbar machen
  4. wenn gewollt downloaden + selbst freigeben

...und das alles am liebsten per BitTorrent

bittorrent: interessant, weil alle kontakte normalerweise nur wenig upload bandbreite haben schnellere Übertragung, je mehr Kontakte eine Datei anbieten xmpp: sowieso den ganzen tag offen -: eigene downloads sind direkt auch immer in der kontaktliste freigegeben

Techniken

# todo: flowchart
  1. BitTorrent: Torrents für freigegebene Dateien/Verzeichnisse erzeugen
  2. XMPP: Torrent Hashes und eigene IP ähnlich der eigenen Statusnachricht an alle Kontakte publishen ("PubSub")
  3. XMPP: bekommene Hashes aggregieren, durchsuchbar machen
  4. BT: neuen Torrent aus Hash adden, Peer-IPs hinzufügen
  5. Kaffee trinken, während BitTorrent sich um den Rest kümmert

Vorteile: - wir brauchen nur die bestehende XMPP infrastruktur - wir sharen immer nur zwischen bekannten Kontakten, unbekannte IPs lassen sich blocken - wir benötigen keinen Torrent Tracker (könnten aber einen benutzen wenn wir wollten -> addiert "normale" BT Funktionen (Peer discovery: DHT, LSD, Tracker...)) -
